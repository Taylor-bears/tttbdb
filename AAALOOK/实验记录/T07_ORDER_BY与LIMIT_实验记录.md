## 2. 题目要求概述

本题要求 RMDB 支持使用 `ORDER BY` 对查询结果进行单列或多列排序。未指定排序方向时默认升序，也应支持显式 `ASC` 和 `DESC`，且多列可以使用不同方向。同时需要实现 `LIMIT`，将排序和过滤后的结果集限制为指定行数，并继续将查询结果写入数据库目录的 `output.txt`。

## 4. 实现方法

### 4.1 修改模块

1. `parser`：增加多列排序列表、每列排序方向和可选 LIMIT 语法。
2. `analyze`：解析排序列所属表，并进行列存在性与歧义检查。
3. `optimizer`：扩展 SortPlan，使计划同时携带多个排序键及 LIMIT。
4. `execution`：完成 SortExecutor 的记录收集、稳定多键比较和结果截断。
5. `portal`：把新的 SortPlan 参数传递到执行器。

### 4.2 实现流程

```text
1. Parser 解析 ORDER BY 列表、ASC/DESC 和可选 LIMIT。
2. Analyze 对每个排序列补全表名并保存方向。
3. Planner 先生成带 WHERE 条件的扫描或连接计划。
4. 在扫描/连接之上建立 SortPlan，再在外层建立 ProjectionPlan。
5. SortExecutor 读取输入记录并按多个键进行稳定排序。
6. 执行器只暴露前 LIMIT 条记录，输出模块写入客户端和 output.txt。
```

### 4.3 关键伪代码

```text
算法：多键排序与 LIMIT

输入：记录列表、排序列列表、方向列表、limit
输出：有序且不超过 limit 的记录流

1. 将子执行器产生的所有记录保存到内存数组。
2. 对两条记录 lhs、rhs，按排序列顺序比较：
   a. 当前列相等时继续比较下一列；
   b. 当前列不等时，根据 ASC/DESC 返回比较结果。
3. 所有键相等时保持原有相对顺序。
4. 游标从排序结果首行开始迭代。
5. 当游标达到 min(记录数, limit) 时结束。
```

## 7. 调试过程记录

| 问题 | 原因 | 解决方法 |
|---|---|---|
| 原有 ORDER BY 查询不返回记录 | 框架中的 SortExecutor 只有空方法，`Next()` 固定返回空指针 | 完整实现加载、排序、迭代和结束判断接口 |
| 原结构只能保存一个排序键 | OrderBy、SortPlan 和 SortExecutor 都使用单列字段 | 将三层结构统一扩展为排序列和方向数组 |
| 按未投影列排序可能找不到字段 | 若先 Projection，源记录中已没有排序列 | 将 SortPlan 放在 ProjectionPlan 之前执行 |
| 多列使用不同方向时结果错误 | 只保存一个全局升降序标记 | 为每个排序列单独保存并应用方向 |
| LIMIT 无 ORDER BY 时仍需工作 | 不能依赖排序键存在才创建执行计划 | 排序键为空时保持输入顺序，只执行行数截断 |

## 8. 可放入实验报告的总结

本题实现了 RMDB 的多列 `ORDER BY` 和 `LIMIT`。解析与语义分析模块负责保存并校验多个排序列及其方向，查询规划器将排序安排在条件过滤之后、最终投影之前，使未投影列也能作为排序键。SortExecutor 采用稳定多键比较，对各列分别应用升序或降序，并在排序完成后限制输出行数。最终题目在希冀平台一次性通过，查询执行、存储管理和解析器回归测试也全部通过。
