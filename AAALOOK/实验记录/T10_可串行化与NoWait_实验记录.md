## 2. 题目要求概述

本题要求系统支持并发事务，通过两阶段封锁保证可串行化隔离级别，并使用 no-wait 策略预防死锁。测试会以多个客户端交错执行事务，检查脏写、脏读、丢失更新、不可重复读和幻读；冲突事务必须立即回滚并向客户端返回 `abort\n`，不能改变原有 SELECT 返回格式。

## 3. 测试点情况

| 测试点 | 测试内容 | 是否通过 | 处理方式 |
|---|---|---|---|
| 脏写 | 两事务并发修改同一表 | 是 | X/X 冲突，后请求立即中止 |
| 脏读 | 读取另一事务未提交修改 | 是 | X/S 冲突，读事务立即中止 |
| 丢失更新 | 两事务基于相同旧值写入 | 是 | 写事务持有 X 锁到结束 |
| 不可重复读 | 两次读取之间发生修改 | 是 | 读事务持续持有 S 锁 |
| 幻读 | 范围查询期间插入或删除 | 是 | 表级 S/X 锁阻止范围变化 |
| 死锁预防 | 冲突请求不等待 | 是 | 抛出事务中止异常并回滚 |

## 4. 实现方法

### 4.1 修改模块

1. `concurrency`：实现锁表、请求队列、兼容矩阵、锁升级和 no-wait。
2. `execution`：顺序扫描和索引扫描在开始时获取共享表锁。
3. `portal`：写语句在扫描或修改之前获取排他表锁。
4. `transaction`：提交和回滚阶段统一释放事务锁集合。
5. `server`：保持冲突事务返回 `abort\n`，并修正隐式事务结束判断。

### 4.2 实现流程

```text
读语句：
1. SeqScan/IndexScan 开始前请求表级 S 锁。
2. 若与其他事务 X 锁冲突，立即抛出中止异常。
3. 否则读取数据，并将 S 锁持有到事务结束。

写语句：
4. Portal 在收集 RID 或执行 INSERT 前请求表级 X 锁。
5. 若存在其他事务 S/X 锁，立即中止当前事务。
6. 否则执行 DML，并将 X 锁持有到事务结束。

事务结束：
7. commit/abort 逐个释放锁，再清空锁集合并更新状态。
```

### 4.3 关键伪代码

```text
算法：no-wait 加锁

lock(txn, object, requested_mode):
    加锁表互斥锁
    if txn 不在 GROWING 阶段:
        abort txn
    if txn 已持有足够强的锁:
        return success
    if requested_mode 与任一其他事务已授予锁不兼容:
        throw TransactionAbortException
    if 是本事务锁升级:
        更新原请求模式
    else:
        添加已授予请求，并加入事务 lock_set
    return success
```

## 7. 调试过程记录

| 问题 | 原因 | 解决方法 |
|---|---|---|
| LockManager 所有方法均直接返回 true | 框架只提供空壳，没有锁表和兼容判断 | 实现受互斥锁保护的请求队列和兼容矩阵 |
| 读后写同一表可能错误自冲突 | 同一事务重复请求或从 S 升级 X 未单独处理 | 查找已有请求，足够强时复用，无其他持有者时升级 |
| UPDATE/DELETE 加锁过晚仍可能丢失更新 | Portal 会先扫描 RID，再构造写执行器 | 在扫描 RID 之前获取表级 X 锁 |
| 范围查询后出现新记录 | 仅锁已读取记录无法覆盖不存在的键 | SELECT 持有表级 S 锁，写事务必须取得 X 锁 |
| 解锁时直接遍历并修改锁集合存在迭代器风险 | unlock 会改变锁表状态 | 先复制事务锁集合，再逐项释放，最后统一 clear |
| 隐式事务中止后又被标记为提交 | 请求循环无条件自动提交非显式事务 | 自动提交前检查事务状态不是 ABORTED/COMMITTED |

## 8. 可放入实验报告的总结

本题实现了基于严格两阶段封锁的可串行化隔离级别。系统为读操作申请表级共享锁，为写操作申请表级排他锁，并将锁持有到事务提交或回滚。LockManager 使用互斥锁保护共享锁表，支持重复请求与锁升级；发生不兼容请求时采用 no-wait 立即中止事务。表级锁还覆盖范围谓词，避免幻读。最终希冀平台一次性全部通过，五类数据异常与前置功能回归均验证正确。
