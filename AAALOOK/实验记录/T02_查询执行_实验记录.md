## 4. 实现方法

### 4.1 修改模块

1. `system`：管理数据库元数据、表文件及句柄生命周期。
2. `analyze`：完成表、字段、条件和值的语义检查。
3. `optimizer`：为扫描、连接和投影生成执行计划。
4. `execution`：实现顺序扫描、嵌套循环连接、投影、更新和删除。
5. 服务入口：捕获语法及语义错误并按要求记录 `failure`。

### 4.2 实现流程

```text
SQL -> Parser 生成 AST
    -> Analyze 校验并生成 Query
    -> Planner 构建扫描/连接/投影计划
    -> Portal 转换为执行算子树
    -> Executor 迭代读取或修改记录
    -> output.txt 写入查询结果或 failure
```

### 4.3 关键伪代码

```text
算法：顺序扫描与条件过滤

1. 从记录文件首个有效槽位开始扫描。
2. 读取当前记录并依次计算 WHERE 条件。
3. 条件全部成立则返回当前 RID 和记录。
4. 否则继续扫描，直到找到匹配记录或到达文件末尾。
```

## 7. 调试过程记录

| 问题 | 原因 | 解决方法 |
|---|---|---|
| `score > 90` 被判类型不兼容 | 浮点字段和整数常量类型不同 | 在语义分析阶段安全转换 `INT` 为 `FLOAT` |
| 多表计划可能出现空子树或无序结果 | 原连接计划依赖被移动的条件和未初始化数组 | 按 SQL 表顺序重建确定性的左深连接树 |
| 语法错误未写入 `failure` | 服务入口只处理了 RMDB 运行时异常 | 对 `yyparse()` 失败分支追加 `failure` |
| 服务启动事务为空指针 | `TransactionManager::begin` 尚未返回事务 | 补充基础事务创建和提交状态维护 |
