## 2. 题目要求概述

本题要求 RMDB 支持显式开启、提交和回滚事务。显式事务只包含增删改查语句，不要求处理并发；平台分别测试有索引和无索引情况下的提交与回滚，其中有索引测试还包含 DATETIME 数据。事务提交后修改应保留，事务中止后所有数据和索引变化都必须恢复到 begin 之前的状态。

## 4. 实现方法

### 4.1 修改模块

1. `transaction`：实现事务状态管理、提交清理和基于写集合的回滚。
2. `execution`：在三类写算子中记录撤销操作所需的 RID 与旧记录。
3. `index` 协作：回滚时根据表索引元数据重新构造联合键并同步恢复索引。
4. `test`：模拟服务器事务生命周期，验证提交、回滚、索引和 DATETIME。

### 4.2 实现流程

```text
事务执行：
1. begin 将当前事务切换为显式模式。
2. INSERT 成功后记录 (INSERT, table, rid)。
3. DELETE 成功后记录 (DELETE, table, rid, old_record)。
4. UPDATE 成功后记录 (UPDATE, table, rid, old_record)。

commit：
5. 释放所有 WriteRecord 和锁，状态改为 COMMITTED。

abort：
6. 从写集合尾部依次取出 WriteRecord。
7. 执行对应逆操作，并同步恢复所有索引键。
8. 清空资源，状态改为 ABORTED。
```

### 4.3 关键伪代码

```text
算法：事务逆序回滚

while write_set 非空:
    write = write_set.pop_back()
    if write.type == INSERT:
        删除当前记录的全部索引键
        删除该 RID 的记录
    else if write.type == DELETE:
        在原 RID 插入 old_record
        插入 old_record 的全部索引键
    else if write.type == UPDATE:
        删除当前记录的全部索引键
        将原 RID 更新为 old_record
        插入 old_record 的全部索引键
事务状态设为 ABORTED
```

## 7. 调试过程记录

| 问题 | 原因 | 解决方法 |
|---|---|---|
| abort 执行后数据没有恢复 | 框架中的 `TransactionManager::abort` 为空实现 | 基于 WriteRecord 实现逆序撤销 |
| 回滚时没有可用的旧记录 | 三类 DML 均未向事务写集合登记操作 | DELETE/UPDATE 保存旧记录，INSERT 保存新 RID |
| 基表恢复但索引查询错误 | 只撤销记录文件，未同步处理 B+ 树索引项 | 按表元数据为每个索引重建键并执行相反操作 |
| 连续修改同一记录时恢复顺序错误 | 若按正序撤销，后续操作依赖的中间状态已被破坏 | 使用栈式写集合，从尾部严格逆序回滚 |
| 隐式事务模式状态不确定 | Transaction 构造时 `txn_mode_` 未初始化 | 构造函数显式初始化为 false，begin 时再切换为 true |

## 8. 可放入实验报告的总结

本题实现了 RMDB 的显式事务控制。系统在执行 INSERT、DELETE、UPDATE 后记录撤销所需信息，commit 负责确认修改并释放事务资源，abort/rollback 按相反顺序恢复记录和全部索引键。该方案支持跨表操作、同一记录连续修改、唯一索引及 DATETIME 数据，同时保留原有单语句隐式事务行为。最终希冀平台一次性全部通过，本地事务专项测试与前置功能回归也全部通过。
